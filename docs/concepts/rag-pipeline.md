# RAG 파이프라인 · HNSW 인덱스

> 사용자의 자연어 질문에 **사내 문서에 근거해**(출처 포함) 답하는 구조. 문서를 잘게
> 잘라 의미 벡터로 저장해두고, 질문과 의미가 가까운 조각을 찾아 LLM에 함께 넘긴다.

## 왜 RAG인가

문서가 수백 페이지면 통째로 LLM에 넣을 수 없고(입력 한계), LLM은 사내 문서를 모른다.
그래서 **질문과 관련된 부분만 찾아서 LLM에 같이 넣어주는** 방식이 RAG
(Retrieval-Augmented Generation)이다.

## 핵심 용어

- **청크(chunk)** = 문서를 잘게 자른 한 조각(문단 몇 개 단위). 통문서를 안 쓰고 자르는 이유:
  1. **의미 선명도** — 임베딩 한 개는 "고정 크기 요약점" 하나라, 통문서를 한 점에 넣으면
     여러 주제가 평균돼 흐려진다. 청크 = 한 주제 = 선명한 점.
  2. **검색 정밀도·출처** — 관련된 문단만 꺼내 LLM에 넘기고, 몇 페이지에서 왔는지 출처를
     표시할 수 있다.
- **질문 벡터(query vector)** = 사용자 질문을 임베딩한 점. 청크 점들과 거리를 비교한다.

## 전체 흐름

```
[인제스트: 문서 넣을 때, 1회]
 문서 → 청크로 자르기 → 각 청크 임베딩 → pgvector(chunks 테이블)에 저장
                                              └ embedding 컬럼에 HNSW 인덱스

[질의: 사용자가 물을 때, 매번]
 질문 → 질문 벡터로 임베딩 → 가장 가까운 청크 N개 검색 → 그 청크들 + 질문을 LLM에 → 답(+출처)
```

`init_db.sql`의 `chunks` 테이블이 이 그림의 저장소다: `embedding VECTOR(1024)`(청크의 점) +
`chunk TEXT`(원문) + `metadata JSONB`(출처). 임베딩·차원·거리척도는 [[embedding-similarity]] 참고.

## HNSW 인덱스 (Hierarchical Navigable Small World)

**푸는 문제:** 청크가 수십만 개면, 질문 벡터와 전부 거리를 재는 완전탐색(brute force)은
너무 느리다(매 질문마다 N번 계산). HNSW는 **전부 안 보고도 가까운 걸 거의 정확히 빨리** 찾는다.

**메커니즘 (두 축):**
1. **그래프 탐색(small world).** 각 점은 가까운 이웃 점 몇 개만 안다. 진입점에서 출발해
   **이웃 중 질문에 더 가까운 점으로 계속 점프**하다, 더 가까운 이웃이 없으면 멈춘다.
   거리가 단조 감소하며 목표에 수렴. 수백만이 아니라 수십 개만 만진다.
2. **계층(hierarchical).** 이웃이 다닥다닥이면 걸음이 작아 먼 거리를 가는 데 홉이 너무
   많아 *느리다*(못 가는 게 아니라 느린 것). 그래서 위층에 점을 적게 두고 **긴 링크(급행)**
   를 얹는다. 검색 = 위층(급행)에서 큰 점프로 동네까지 빠르게 → 아래층(완행)에서 정밀화 →
   맨 아래(전 점)에서 최근접 확정.

**Approximate(ANN)인 이유:** 모든 점을 안 보고 한 경로의 이웃만 탐험하므로, 진짜 1등이
안 들른 경로에 있으면 **가끔 놓친다**. 맞힌 비율 = **recall**. 파라미터로 recall을 올리면
정확↑·속도↓ (속도↔정확 다이얼). 즉 "틀린/랜덤"이 아니라 **거의 정답 + 작고 조절 가능한
오차**를 수십~수백배 속도와 맞바꾼 것.

**규모 감각:**
```
청크 30만, 1024차원, 코사인:
  완전탐색  = 매 질문 300,000번 거리 계산
  HNSW     ≈ log(N) 비례 → 수백 번 수준  (수백 배 빠름)
```

**pgvector 매핑:** `CREATE INDEX ... USING hnsw (embedding vector_cosine_ops)` =
"`embedding` 컬럼에 코사인 거리 기준 HNSW 그래프를 깔아라." → `retrieval.py`의 검색
연산자도 코사인(`<=>`)이어야 일치한다. (대안 `ivfflat`: 군집 기반, 빌드 빠름·메모리↓,
검색은 보통 HNSW가 우위)
