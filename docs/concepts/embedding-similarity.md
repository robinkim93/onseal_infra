# 임베딩 · 벡터 차원 · 유사도

> 텍스트를 "의미의 좌표(숫자 리스트)"로 바꾸고, 그 좌표들이 **얼마나 가까운지**로 의미 유사도를 재는 것. RAG 검색의 토대.

## 직관 (왜 이렇게 생겼나)

임베딩 모델은 **텍스트 한 조각 → 고정 길이의 숫자 리스트**로 변환한다.

```
"부가가치세 신고"  ──[임베딩 모델]──>  [0.21, -0.83, 0.05, ... , 0.44]
                                       └────── 숫자 N개 ──────┘
```

이 리스트는 그 문장을 **N차원 공간의 한 점**으로 찍은 것이다. 의미가 비슷한 텍스트는
공간에서 **가까운 점**이 된다.

```
질문점 ●
        ● ← 가까움 = 의미 비슷 (검색 결과로 뽑힘)
                      ● ← 멀다 = 무관
```

## 메커니즘 (어떻게 동작하나)

1. **차원 N** = 그 숫자 리스트의 길이. **모델마다 고정**돼 있다(예: 768, 1024). 내가
   정하는 게 아니라 **고른 모델이 정한다.**
2. **검색** = 질문을 점으로 찍고, 저장된 청크 점들 중 **제일 가까운** 것들을 찾는 일.
3. **거리 척도** = "가깝다"를 재는 자. 대표적으로:
   - **코사인(cosine)** — 두 벡터의 *방향* 유사도 (크기 무시). 텍스트 임베딩 기본값.
   - **유클리드(L2)** — 두 점 사이 직선 거리.

## 이 프로젝트에서 왜 중요한가 (Onseal / pgvector)

- pgvector 컬럼은 `embedding VECTOR(N)`처럼 **차원 N을 고정**해야 한다.
- 그래서 **임베딩 모델을 먼저 골라야** N이 정해지고, 그제야 테이블을 만들 수 있다.
  (M1의 숨은 의존성: 모델 선정 → `VECTOR(N)` → `init_db.sql` 완성)
- 검색 인덱스의 **연산자 클래스**(`vector_cosine_ops` / `vector_l2_ops`)는 `retrieval.py`가
  쓸 거리 척도와 **반드시 일치**해야 한다. 코사인으로 저장·인덱스했으면 검색도 코사인.
- 에어갭 제약: 임베딩은 **로컬 모델**만. 외부 임베딩 API 금지.

## 흔한 함정

- **차원은 "성능 힌트"가 아니라 "절대 일치해야 하는 계약"이다.** `VECTOR(768)` 컬럼에
  1024차원 벡터를 넣으면 *틀린 유사도*가 아니라 **INSERT 자체가 거부**된다
  (`ERROR: expected 768 dimensions, not 1024`). 길이가 다른 두 벡터는 거리 정의 자체가
  성립하지 않기 때문.
- **모델을 바꾸면 차원이 바뀔 수 있다** → 기존 테이블·벡터를 재생성(re-embed)해야 한다.
- **인덱스 연산자 불일치** → 인덱스가 무시되거나 엉뚱한 결과. 척도를 한 곳에서 통일할 것.
- **SQL 파일은 `${}` 치환을 안 한다.** `VECTOR(EMBEDDING_DIM)`은 문법 에러 — 진짜 숫자를
  박아야 한다(compose의 환경변수 치환과는 다른 층위).
